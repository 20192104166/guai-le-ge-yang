## 1. 通用原则

### 1.1 可读性
- 命名优先于注释：让代码自解释
- 单个函数 ≤ 50 行，单文件 ≤ 500 行（合理拆分）
- 避免魔法数字，提取为命名常量

### 1.2 命名
- 文件 / 类：PascalCase（如 `ShagaiPhysics`）
- 函数 / 变量：camelCase（如 `flickShagai`）
- 常量：UPPER_SNAKE_CASE（如 `MAX_SHAGAI_COUNT`）
- 私有：前缀 `_`
- 不允许拼音命名（如 `yangguai`）→ 用 `shagai`

### 1.3 注释
- 公共 API 必须有 JSDoc 注释
- 复杂算法上方写"为什么"，不是"做什么"
- TODO / FIXME 必须带责任人与日期

### 1.4 错误处理
- 不允许吞异常（`catch` 空块）
- 错误必须 log 或上抛
- 用户能看到的错误必须友好提示

## 3. 游戏代码组织（暂定）

```
src/
├── core/        基础设施（场景管理、事件总线、配置加载）
├── physics/     物理相关（羊拐物理、形态判定）
├── gameplay/    玩法逻辑
│   ├── flick/   弹沙嘎
│   ├── toss/    抛沙嘎
│   └── catch/   抓沙嘎
├── ui/          UI 组件
├── data/        数据层（持久化、本地配置）
├── platform/    抖音 SDK 封装
└── utils/       工具函数
```
