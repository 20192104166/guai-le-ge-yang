### 2.1 `_Project/` 下划线置顶的原因
Unity 默认按字母排序，`_Project/` 排在最上方，与 `Plugins/`、`Editor/` 等系统目录视觉隔离。**所有自研内容入 `_Project/`**。

### 2.2 哪些目录纳入 Git？

| 目录 | Git | 说明 |
|---|---|---|
| `Assets/` | ✅ | 必须 |
| `Packages/` | ✅ | manifest.json + lock 文件 |
| `ProjectSettings/` | ✅ | 工程设置 |
| `Library/` | ❌ | Unity 缓存，重建即可 |
| `Temp/`、`Logs/` | ❌ | 临时 |
| `UserSettings/` | ❌ | 个人偏好 |
| `obj/`、`Build*/` | ❌ | 构建产物 |

### 2.3 .meta 文件规则
- **每个 Assets/ 下的文件 / 目录都有对应的 .meta 文件**
- `.meta` 必须随主文件一起 commit
- 删除文件时也要删 .meta
- 移动文件时保留同名 .meta（避免引用断裂）

## 四、Assembly Definition（程序集划分）

Unity 默认所有 Scripts 在 `Assembly-CSharp` 一个程序集中，编译慢。建议拆分：

```
_Project/Scripts/
├── GuaiLeGeYang.Core.asmdef            ← 无依赖
├── GuaiLeGeYang.Physics.asmdef         ← 依赖 Core
├── GuaiLeGeYang.Data.asmdef            ← 依赖 Core
├── GuaiLeGeYang.Platform.asmdef        ← 依赖 Core + Data
├── GuaiLeGeYang.Gameplay.asmdef        ← 依赖 Core + Physics + Data
├── GuaiLeGeYang.UI.asmdef              ← 依赖 Core + Data + Gameplay
└── GuaiLeGeYang.Utils.asmdef           ← 无依赖
```

收益：
- 改一个文件仅重编对应程序集，编译时间 ↓
- 强制依赖方向（防止环形依赖）
- 单元测试方便（每个 asmdef 配一个 Tests）

> Phase 1 可暂不拆分 asmdef，先开发；Phase 2 进入 MVP 时再拆。

## 六、Resources / StreamingAssets / Addressables

| 加载方式 | 用途 | 限制 |
|---|---|---|
| **直接引用**（Inspector 拖入）| 默认方式，最稳 | 必须打包进场景 / Prefab |
| `Resources.Load` | 运行时按名加载 | ⚠️ 进主包，节制使用 |
| `StreamingAssets/` | 配置文件、可热更新数据 | 不被 Unity 压缩；按平台行为 |
| `Addressables` | 资源按需加载（CDN）| 复杂但灵活，**抖音小游戏适配性需 Phase 1 验证** |

### 推荐策略（抖音小游戏）
- 启动时必需资源：直接引用 / Resources
- 后续按玩法加载：分包（抖音小游戏分包机制）+ StreamingAssets
- 美术热更新：CDN + 抖音云存储
- **不强求 Addressables**（在 WebGL/抖音环境兼容性需验证）
