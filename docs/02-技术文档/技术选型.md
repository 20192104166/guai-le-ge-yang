## 1. Unity 技术栈细节

### 1.1 Unity 版本
- **推荐版本**：Unity 2022 LTS（兼容性好、抖音小游戏官方支持）
- **次选**：Unity 2021 LTS（更稳定）
- **避免**：Unity 6 / 最新版（抖音平台尚未充分验证）

### 1.2 抖音小游戏适配方案
Unity 发布到抖音小游戏有两条主流路径：

| 方案 | 说明 | 优劣 |
|---|---|---|
| **Unity WebGL → 抖音小游戏适配** | 用抖音官方提供的 Unity WebGL 适配工具 | 包体较大（10MB+），需做分包 |
| **Unity Tiny → 抖音** | Unity Tiny 输出极小包体 | Tiny 已停更，不建议 |
| **Unity for Mini Games（字节官方插件）** | 字节官方 Unity 转换工具 | **推荐**，原生体验 |

> ⚠️ 待 Phase 1 时由开发同学实测包体大小与性能

### 1.3 编程语言
- **C#** （Unity 默认）
- 启用 **.NET Standard 2.1**

## 2. 2D 物理引擎

Unity 内置 **Box2D**（2D Physics），无需额外集成。

### 2.1 不规则形状处理（核心问题）
羊拐为不规则物体，Box2D 支持：
- `PolygonCollider2D`（多边形碰撞体）：手绘多边形外形
- `CompositeCollider2D`：多形状组合

### 2.2 物理实现混合策略
由于 2D 物理无法天然表现 3D 翻滚的不等概率，采用 **物理 + 加权随机 混合**：

```
玩家操作 → 2D 物理模拟翻滚动画（视觉真实感）
              ↓
         物理静止时
              ↓
   按设定概率分布随机决定最终朝上面（保证公平性可控）
              ↓
        播放"翻转就位"动画过渡到最终面
```

**优点**：
- 视觉上有真实物理感
- 概率可控（避免某面长期不出导致体验问题）
- 性能开销小

**风险**：
- 玩家可能察觉"伪物理" → 需要美术做"翻转就位"动画掩盖
- Phase 1 必须验证体感是否自然

详细方案见 [`物理引擎方案.md`](物理引擎方案.md)。

## 3. ✅ 后端方案（已确认：抖音云开发）

由于需要 **PvP 异步 + 排行榜 + 图鉴收集 + 好友数据**，必须引入服务端。**用户已确认采用抖音云开发**。

### 3.1 候选方案对比

| 方案 | 开发成本 | 运营成本 | 灵活性 | 与抖音生态契合度 |
|---|---|---|---|---|
| **A. 抖音云开发**（推荐） | ⭐ 低 | ⭐ 低 | ⭐⭐ 中 | ⭐⭐⭐ 高 |
| B. 自建服务器 | ⭐⭐⭐ 高 | ⭐⭐⭐ 高 | ⭐⭐⭐ 高 | ⭐ 低 |
| C. BaaS（LeanCloud 等） | ⭐⭐ 中 | ⭐⭐ 中 | ⭐⭐ 中 | ⭐ 低 |

### 3.2 抖音云开发能力
- **云函数**：无服务器函数（JavaScript/Node.js）
- **云数据库**：MongoDB 兼容文档数据库
- **云存储**：图片、资源
- **实时通讯**：内置 WebSocket 能力（用于实时 PvP）
- **认证**：与抖音账号一键打通
- **CDN**：与抖音同源加速

### 3.3 ✅ 决策：抖音云开发
- 最贴近平台生态
- 接入抖音登录 / 分享最顺
- 初期成本几乎为零（赠送额度）
- 客户端 C# + 云函数 Node.js 双语言协作

> 详细的云数据库 schema、云函数列表、客户端 SDK 接入将在 **`服务端架构.md`** 中展开（Phase 0 末尾编写）。

## 4. ✅ PvP 实现方案（已确认：V1.0 仅异步）

### 4.1 异步 PvP（推荐先做）
- 玩家 A 完成一局，记录"操作录像"到云端
- 玩家 B 点击对战，下载录像本地回放，再操作自己一局
- 双方各自录像 + 结果上传，按得分算胜负

**优点**：
- 无需实时连接 / 房间匹配
- 容错好（断网无影响）
- 服务端压力小（只存录像）
- 开发量小

**缺点**：
- 不是"真对战"，缺少即时感

### 4.2 实时 PvP（可后续加）
- WebSocket 长连接
- 匹配系统
- 状态同步 / 帧同步
- 断线重连处理

**优点**：
- 真竞技体验

**缺点**：
- 开发复杂
- 服务器压力大
- 抖音用户网络环境多变，体验不稳

### 4.3 ✅ 决策路径
**V1.0 只做异步 PvP**，V1.1+ 视用户量与反馈决定是否补充实时 PvP。

实施要点：
- 三玩法（弹/抛/抓）的"局录像"格式需统一抽象（操作序列 + 随机种子 + 关键事件）
- 录像上传到云存储；回放在客户端进行
- 双方录像合并后由云函数裁定胜负（不在客户端做权威判断）

## 6. 包体策略

抖音小游戏主包限制约 **4MB**，分包可加大但有总限。优化方向：

| 类别 | 策略 |
|---|---|
| 美术资源 | 走 CDN / 分包加载，主包只留必要 |
| 音频 | OGG 格式 + 流式加载 |
| 字体 | 仅打包用到的字符（中文字库子集化） |
| 代码 | C# 程序集裁剪、代码剥离 |
| 第三方库 | 谨慎引入，能用 Unity 自带绝不引外部 |
